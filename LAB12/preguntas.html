Q & A

  • ¿Por qué es importante hacer un session_unset() y luego un session_destroy()?
    Es importante porque session_unset() solo se encarga de limpiar la variable $_SESSION, por lo tanto, sólo afecta de manera local y no la información de sesión guardada. Es por eso que se debe hacer un session destroy para que la información guardada también sea destruida, removiendo así la sesión por completo. [1]
  • ¿Cuál es la diferencia entre una variable de sesión y una cookie?
    Una cookie es un fragmento de información que un navegador web almacena en el disco duro del visitante a una página web. La información se almacena a petición del servidor web y puede ser recuperada por el servidor web en posteriores visitas a la misma página.

    Gracias a las cookies se puede compartir información entre distintas páginas de un sitio web o incluso en la misma página web pero en diferentes instantes de tiempo, resolviendo así el problema del protocolo HTTP siendo el sin estado.

    Una sesión es un mecanismo de programación de las tecnologías de web scripting que permite conservar información sobre un usuario al pasar de una página a otra. A diferencia de una cookie, los datos asociados a una sesión se almacenan en el servidor y nunca en el cliente. [2]
  • ¿Qué técnicas se utilizan en sitios como facebook para que el usuario no sobreescriba sus fotos en el sistema de archivos cuando sube una foto con el mismo nombre?
    Se verifica la existencia de un archivo de mismo nombre antes de moverlo al directorio correspondiente, si no existe se guarda con el nombre sin modificar, pero cuando si existe se renombra con un infijo numérico.
  • ¿Qué es CSRF y cómo puede prevenirse?
    CSRF (acrónimo de *Cross-site request forgery*) es un método por el cual un usuario malintencionado intenta hacer que tus usuarios, sin saberlo, envíen datos que no quieren enviar. Para evitar estas situaciones es necesario añadir un *token* CSRF (campo oculto) a tus formularios.[4]
