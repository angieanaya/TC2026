Información adicional sobre las tecnologías

    El esquema del URL es como el siguiente:
    //host.dominio:puerto/directorio/nombre_de_archivo

    MIME usa directivas especiales en los encabezados para describir el formato utilizado en el cuerpo de un mensaje.
    El esquema está basado en lo siguiente:
    Tipo de contenido: tipo_mime_principal/subtipo_mime

    1XX - Indica una respuesta provisional.
    100 Continue. El servidor ha recibido los headers del request y el cliente debería proceder a enviar el cuerpo de la respuesta.
    101 Switching Protocols. El requester ha solicitado al servidor conmutar protocolos.
    102 Processing. Usado en requests para reanudar peticiones PUT o POST abortadas.

    2XX - Indica que la acción ha sido exitosa.
    200 OK. El request es correcto.
    201 Created. Se ha creado un nuevo recurso.
    202 Aceptada. El proceso del request aún no ha terminado.
    203 Non-Authoritative Information. El request devuelve información que podría venir de otra fuente.
    204 No Content. El request no devuelve ningún contenido.
    205 Reset Content. El request no devuelve ningún contenido y se requiere que el requester recargue el contenido.
    206 Partial Content. El servidor devuelve sólo parte del recurso debido a una limitación que ha configurado el cliente.

    3XX - El cliente ha de tomar una acción adicional para completar el request.
    300 Multiple Choices. Es una lista de enlaces con máximo de cinco direcciones.
    301 Moved Permanently. La página solicitada se ha movido permanentemente a una nueva URL.
    302 Found. La página solicitada se ha movido temporalmente a una nueva URL.
    303 See Other. La página solicitada se puede encontrar en una URL diferente.

TECNOLOGÍAS PARA EL DESARROLLO WEB

URL (Uniform Source Locator)

Es la dirección única y específica que se asigna a cada uno de los recursos que se encunetran disponibles en WWW para que estos puedan ser localizados por el navegador y desplegados ante el usuario. La dirección provee al navegador de diferentes datos que le indican los pasos a seguir para llegar hasta el recurso solicitado.

MIME (Multipurpose Internet Mail Extensions)

Estandár propuesto en 1991 por Bell Communications para expandir las diferentes capacidades del mismo, a través de longitud ilimitada de texto, capacidad de adjuntar varios documentos, uso de caracteres no pertenecientes a ASCII, texto enriquecido y capacidad de adjuntar ejecutables, imagenes y/o archivos de audio y video.

Métodos y códigos de errores HTTP

El uso de métodos es obligatorio para realizar solicitudes al servidor, estos estñan pre definidos y le indican al servidor la acción a realizar.

En cuanto a los codigos de errores, estos fueron especificados por el RFC 2616 y cuenta con 5 tipos de respuesta.

    1XX - Respuestas informativas
    2XX - Peticiones correctas
    3XX - Redirecciones
    4XX - Errores del cliente
    5XX - Errores de servidor

HTML5 (Hypertext Markup Language)

Quinta versióin del lenguaje markup usado para estructurar y presentar el contenido para la web.

JS (JavaScript)

Lenguaje de programación sencillo creado por Brendan Eich en 1995 (LiveScript) destinado a utilizarse para el diseño de una página interactiva, convirtiéndose asi en el lenguaje de programación que se utiliza en casi todos los sitios web en el mundo.

php (Hypertext Preproccesor)

Lenguaje de código abierto creado en 1995 por Rasmus Lerdorf el cual ofrece una salida de texto con codificación UTF-8 compatible con los documentos HTML. PHP puede ser desplegado en la mayoría de los servidores web y en cualquier sistema operativo sin costo alguno.

jsp (Java Server Pages)

Lenguaje desarrollado por Sun Microsystems orientado a desarrollar páginas web en Java y creado para ejecutarse del lado del servidor. Esta tecnología permite a los desarrolladores desarrollar rápidamente y mantener de manera sencilla páginas dinámicas, ricas en información como son las que soportan a sistemas de negociación.

Ajax (Asynchronous JavaScript And XML)

Técnica de desarrollo web para crear aplicaciones interactivas o RIA(Rich Internet Applications). Estas aplicaciones se ejecutan en el cliente, es decir, en el navegador de los usuarios mientras se mantiene la comunicación asíncrona con el servidor en segundo plano. De esta forma es posible realizar cambios sobre las páginas sin necesidad de recargarlas, mejorando la interactividad, velocidad y usabilidad en las aplicaciones.

PREGUNTAS Y RESPUESTAS

¿Cuál es la diferencia entre Internet y la World Wide Web?

Internet es una red global de redes mientras que la WWW es un recinto de información al cual se accede a través del Internet.

¿Cuál es el propósito de los métodos HTTP: GET, HEAD, POST, PUT, PATCH, DELETE?

GET Solicita la representación de un recurso específico(recupera datos)
HEAD Solicita una respuesta idéntica a la de una petición GET, pero sin el cuerpo de la respuesta.
POST Envía una entidad a un recurso en específico, causando así cambios en el estado del servidor.
PUT Reemplaza todas las representaciones actuales del recurso de destino con la carga útil de la petición.
PATCH Aplica modificaciones parciales a cualquier recurso.
DELETE Borra un recurso específico

¿Qué método HTTP se debe utilizar al enviar un formulario HTML, por ejemplo cuando ingresas tu usuario y contraseña en algún sitio? ¿Por qué?

Se debe utilizar el método POST pues se trata de información sensible.

¿Qué método HTTP se utiliza cuando a través de un navegador web se accede a una página a través de un URL?

Se debe utilizar el método GET pues es específico para peticiones de información que no es sensible

Un servidor web devuelve una respuesta HTTP con código 200. ¿Qué significa esto? ¿Ocurrió algún error?

El código 200 indica que el request no ha producido ningún error, es la respuesta estándar para respuestas correctas.

¿Es responsabilidad del desarrollador corregir un sitio web si un usuario reporta que intentó acceder al sitio y se encontró con un error 404? ¿Por qué?

Si el error no fue del cliente a la hora de teclear el url de la página entonces lo más probable es que el link no está activo o actualizado y es responsabilidad del desarrollador resolver dicha problemática. De no ser así, el cliente sería responsable.

¿Es responsabilidad del desarrollador corregir un sitio web si un usuario reporta que intentó acceder al sitio y se encontró con un error 500? ¿Por qué?

Si, porque el error que arroja hace referencia a un problema interno en el servidor y en este caso el programa que el desarrollador mandó al servidor está mal.

¿Qué significa que un atributo HTML5 esté depreciado o desaprobado (deprecated)? Menciona algunos elementos de HTML 4 que en HTML5 estén desaprobados.

Significa que son atributos que la W3C recomienda dejar de utilizar pues existen alternativas más efectivas, sin embargo manteniendo aún así soporte para sitios que ocupen dichos atributo y así no se pierda compatibilidad con sitios antiguos.

    Etiquetas: acronym, applet, center, big, fotn, frame, strike, u, tt, s y otros.
    Atributos: shape, target, version, name, scheme, type, align, link, bgcolor, background, cellpadding, cellspacing, width y otros.

¿Cuáles son las diferencias principales entre HTML 4 y HTML5?

La principal diferencia en HTML5 es que se separa en la estructura el contenido, la presentación (CSS) y el comportamiento (JS).

Además, es m´ss simple, cuenta con un mejor manejo de errores y por lo tanto provee mayor consistencia en documentos de formato incorrecto.

Más aún, introduce nuevas etiquetas como: canvas, audio y video soportando de esta manera archivos multimedia por sí solo.

¿Qué componentes de estructura y estilo tiene una tabla?

    Estilo: border-collapse, border-spacing, padding, text-align, border, font-weight, background, white-space, border-color and margin.

Input, form, datalist y output.

Sobre el ciclo de vida y desarrollo de los sistemas de información:

¿Cuál es el ciclo de vida de los sistemas de información? ¿Cuál es el ciclo de desarrollo de sistemas de información?
1. Planificación

Definir el problema, identificar áreas de oportunidad/soluciones, generar objetivos del sistema y delimitar el alcance del proyecto.

1. Definición de requerimientos

Análisis de las necesidades del cliente, determinar características principales del sistema por medio de la descripción de requisitos que servirá como base para estimar costo, tiempo y alcance.

2. Análisis

Definir los requisitos funcionales y no funcionales del sistema. Identificar reglas de negocio y definir un plan de trabajo para el desarrollo del sistema tomando en cuenta los riesgos que se puedan presentar, así como la selección de un sistema de versionado.
2. Análisis y diseño de software

Se describe la estructura interna del software y las relaciones entre las entidades que lo componen.
3. Diseño

Descripción detallada de los requisitos del sistema que servirá como base para la siguiente etapa.

3. Implementación y prueba de unidades

Basado en la parte de diseño se empieza a programar, por lo que se debe elegir el lenguaje que mas se adecue al problema y dedinir una manera de versionar el código.
4. Implementación

Codificar lo descrito en los casos de uso, implementando patrones de diseño si es posible y manteniendo un codigo cohesivo y comentado para poder ver los avances que se van logrando. Generar la base de datos.

4. Integración y prueba del sistema

Verificar que todos los componentes del sistema funcionen de manera correcta y cumplan con las necesidades planteadas.
5. Pruebas

Detectar errores o casos de uso no validados dentro del sistema.
5. Operación y mantenimiento

Se instala y se verifica el funcionamiento correcto del software en su entorno definitivo. Dar mantenimiento para corregir errores o mejorar el rendimiento.

6. Instalación/despliegue

Planificar el entorno en el que el sistema puede/debe funcionar tomando en cuenta tanto el hardware como el software.
7. Uso y mantenimiento

Identificar los defectos detectados durante su vida útil, adaptarlo a nuevas necesidades y añadirle funcionalidades que impliquen una mejora en este.
